Analyse et proposition d'architecture

  Ce qu'on a aujourd'hui

  Le Client a déjà :
  - typeClient: TypeClient? → COMPTANT | CREDIT (mode de paiement, pas un
  segment)
  - etat: MemberStatus → lifecycle (ACTIF, SUSPENDU, etc.)
  - Aucune notion de segment / catégorie (RIA vs ordinaire)
  - Aucun système de tags

  ---
  Ce que je propose

  2 ajouts, 100% additifs, zéro rupture sur les données existantes :

  ---
  1. Champ segment sur Client

  enum SegmentClient {
    ORDINAIRE
    RIA
  }

  model Client {
    ...
    segment  SegmentClient  @default(ORDINAIRE)  // tous les existants →
  ORDINAIRE
    ...
  }

  - Migration : ALTER TABLE "Client" ADD COLUMN "segment" ... DEFAULT
  'ORDINAIRE' — safe en prod, Postgres applique le default sur toutes les lignes
   existantes sans downtime
  - Permet des filtres/index propres (WHERE segment = 'RIA')
  - Extensible (ajouter VIP, GROSSISTE… plus tard = juste un addValue Prisma)

  ---
  2. Système de tags (Tag + ClientTag)

  model Tag {
    id          Int      @id @default(autoincrement())
    nom         String   @unique
    couleur     String   @default("#6366f1")  // hex pour l'UI
    description String?
    // Optionnel : restreindre un tag à un segment
    segment     SegmentClient?  // null = applicable à tous
    actif       Boolean  @default(true)

    createdAt   DateTime @default(now())
    clients     ClientTag[]
  }

  model ClientTag {
    clientId  Int
    tagId     Int
    createdAt DateTime @default(now())

    client    Client  @relation(fields: [clientId], references: [id], onDelete:
  Cascade)
    tag       Tag     @relation(fields: [tagId], references: [id], onDelete:
  Cascade)

    @@id([clientId, tagId])  // pas de doublons
  }

  - Tags gérés par l'admin (CRUD)
  - Chaque tag peut être "universel" ou réservé à un segment (ex: le tag "Membre
   fondateur" n'a de sens que pour RIA)
  - La suppression d'un client cascade sur ses tags (onDelete: Cascade)
