 Workflow Inter-Commissions RIA — finaliser le cahier de charge

 Contexte

 Le module Gouvernance RIA a déjà son squelette de données pour les dossiers
 inter-commissions (DossierInterCommission, VersionDossierIC,
 EchangeInterCommission, enums StatutDossierInterCommission et
 TypeDossierIC créés "CDC"), avec une API admin de base
 (/api/admin/ria/commissions/gouvernance/dossiers + [id] + [id]/echanges).//
 Mais en l'examinant pour implémenter les 4 scénarios du cahier de charge que
 l'utilisateur vient de coller, plusieurs trous et incohérences bloquent le
 workflow réel :

 - Les pages dossiers/page.tsx (admin et portail responsablesRIA) ont été
 écrites contre d'anciennes valeurs d'enum qui n'existent plus dans le schéma
 (BROUILLON/AJUSTEMENT au lieu de EN_PREPARATION/EN_ATTENTE_DECISION,
 types DEMANDE_AVIS/SAISINE/... au lieu de DEMANDE_FINANCEMENT/...) et
 référencent des champs priorite/dateEcheance absents du modèle. Créer un
 dossier ou filtrer par statut échoue silencieusement ou plante.
 - Aucune page de détail ([id]/page.tsx) n'existe ni côté admin ni côté
 portail — impossible d'ouvrir un dossier, de voir son historique de
 versions/échanges, ou de déclencher une action de workflow depuis l'UI.
 - Les membres de commission (Président, Rapporteurs) n'ont aucun moyen
 d'agir : /api/membreCommission/dossiers n'a qu'un GET en lecture. Toutes
 les actions de workflow (PATCH [id], POST [id]/echanges) sont
 verrouillées derrière getRIASession (Admin/SuperAdmin/RESPONSABLE_RIA
 uniquement) — donc dans la pratique, seule la Direction Générale peut faire
 fonctionner le workflow, alors que le cahier de charge décrit explicitement
 la Commission Opérations et la Commission Finance comme actrices.
 - La transition DEMANDER_AJUSTEMENT ramène au statut TRANSMIS au lieu de
 EN_PREPARATION — empêche la Commission Opérations de modifier le dossier
 avant de le retransmettre (Scénario 3).
 - Aucun calcul automatique (ROI, taux de risque, etc. — Scénario 2) ni
 création automatique de financement à l'exécution (Scénario 4).
 - Pas de vue "Historique des échanges" globale (item du sous-menu demandé).

 Note : en explorant, j'ai aussi trouvé que la page de détail "Commission
 Audit & Contrôle" plante partout dans le module (22 fichiers envoient
 "AUDIT_CONTROLE" alors que l'enum réel est "AUDIT"). Hors scope ici —
 sera traité comme tâche séparée plus tard.

 Décisions actées avec l'utilisateur

 - Le bug AUDIT_CONTROLE est hors scope de cette tâche.
 - APPROUVE et EXECUTE restent deux statuts distincts (décision Finance,
 puis décaissement effectif). La création du financement (autorisation de
 décaissement + affectation portefeuille + mouvement de fonds) est
 déclenchée par l'action EXECUTER, pas par APPROUVER.
 - Les calculs automatiques (Scénario 2) utilisent les vraies données :
 Client.scoreSolvabilite pour taux de risque / probabilité de
 remboursement, PortefeuilleRIA.capitalDisponible pour l'impact
 trésorerie.

 Modèle de données — structure du contenu JSON

 Les champs métier listés dans le Scénario 1 (région, agence, liste de
 clients, produits, coûts, marge, etc.) ne deviennent pas des colonnes : ils
 vivent dans VersionDossierIC.contenu (Json), qui existe déjà pour
 versionner les révisions (Scénario 3 — Versions 1 à 4). Forme pour
 type = DEMANDE_FINANCEMENT :

 {
   region: string; agence: string; responsableDemandeurId: number;
   clients: { clientId: number; nom: string; montant: number;
              produits: { nom: string; quantite: number; coutAchat: number;
 prixRevente: number }[] }[];
   dureeCycleJours: number;
   risqueEstime: "FAIBLE" | "MOYEN" | "ELEVE"; // saisi par Opérations
   investisseursConcernes: number[]; // ids PortefeuilleRIA candidats
   piecesJointesUrls: string[];
 }

 Seul ajout de colonne nécessaire : portefeuilleExecutionId Int? sur
 DossierInterCommission (+ relation vers PortefeuilleRIA) — le portefeuille
 choisi par le Président Finance au moment d'APPROUVER, utilisé ensuite par
 EXECUTER pour créer les financements.

 Plan d'implémentation

 1. Schéma (prisma/schema.prisma)

 Ajouter portefeuilleExecutionId Int? + relation portefeuilleExecution
 PortefeuilleRIA? sur DossierInterCommission. L'utilisateur gère la
 migration lui-même.

 2. Logique de workflow partagée (lib/dossierInterCommissionWorkflow.ts)

 Extraire la logique actuelle de
 app/api/admin/ria/commissions/gouvernance/dossiers/[id]/route.ts (PATCH)
 dans une fonction appliquerActionDossier(tx, { dossierId, userId, action,
 montantApprouve, portefeuilleExecutionId, contenuRevise, motifRevision,
 commentaire, ... }) réutilisable par l'API admin et la nouvelle API
 membreCommission. Corrections incluses :
 - DEMANDER_AJUSTEMENT → statut EN_PREPARATION (au lieu de TRANSMIS).
 - EXECUTER : si dossier.type === "DEMANDE_FINANCEMENT", lit
 contenu.clients[] de la version courante + portefeuilleExecutionId,
 puis pour chaque client crée un OperationFinancementRIA en réutilisant
 exactement le pattern de app/api/admin/ria/financements/route.ts POST
 (vérif. capitalDisponible, décrément/incrément portefeuille, création
 MouvementFondsRIA, appel à ecritureFinancementRIA de
 lib/riaComptable.ts). Rejette si capital insuffisant.
 - Gating Président via isPresident() (déjà dans
 lib/authCommissionRIA.ts) : TRANSMETTRE requiert président de
 commissionEmettrice ; APPROUVER/REJETER/DEMANDER_AJUSTEMENT/
 EXECUTER requièrent président de commissionReceptrice. Les autres
 actions (VALIDER_RECEPTION, METTRE_EN_ANALYSE, METTRE_EN_ATTENTE)
 sont ouvertes à tout membre actif de la commission concernée. Admin/
 RESPONSABLE_RIA passent toujours.

 3. Calcul automatique (lib/riaAnalyseDossier.ts)

 calculerAnalyseFinancement(contenu, portefeuille) retourne :
 - roiEstime (%) = marge totale (Σ (prixRevente − coûtAchat) × quantité) / Σ
 montant × 100
 - tauxRisque (%) = 100 − moyenne(Client.scoreSolvabilite) des clients listés
 - probabiliteRemboursement (%) = moyenne(Client.scoreSolvabilite)
 - impactTresorerie (%) = Σ montant / portefeuille.capitalDisponible × 100
 - rentabiliteAttendue (FCFA) = Σ montant × roiEstime / 100

 Appelé depuis le GET détail (admin + membreCommission) dès que
 statut ∈ {EN_ANALYSE, EN_ATTENTE_DECISION, APPROUVE, EXECUTE} et que le
 dossier a un portefeuilleExecutionId ou au moins un investisseur concerné
 défini.

 active de l'auteur (commissionEmettrice = sa commission, pas modifiable
 côté client), statut EN_PREPARATION, version 1. Réutilise le pattern de
 création déjà dans l'API admin.
 - GET/PATCH /api/membreCommission/dossiers/[id] — GET inclut l'analyse
 calculée ; PATCH appelle appliquerActionDossier avec le gating ci-dessus.
 - POST /api/membreCommission/dossiers/[id]/echanges — même validation que
 la route admin, auteur = utilisateur courant.
